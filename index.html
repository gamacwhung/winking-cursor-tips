💡 給劉思源

1. 跨出「只分析」的舒適圈，多讓 Cursor 幫你寫

你的人工把關習慣很值得肯定，但目前用法停留在「請 AI 分析」可能可惜了 Cursor 的能力。其實 Cursor 本來就有完整的「先預覽再決定」機制，可以放心讓 AI 動手：

Cursor 內建的安全機制：
  • 整段修改：讓 Cursor 直接改，每個 diff 都會跳出 Accept / Reject 按鈕，不滿意按 Reject 或 Ctrl+Z 就回到原狀，沒有風險
  • 局部片段修改：在編輯器裡用滑鼠選取要改的程式碼 → 按 Ctrl+K → 用自然語言下指令（例如「改成 async」「加 null 檢查」「改用 LINQ」），可以在小範圍實驗不影響其他地方
  • 整檔重構：在 Chat 用 @檔名 加上具體目標，例如：
    • 「把這個類別的職責拆成 Manager + System 兩層」
    • 「事件訂閱補上對應的 unsubscribe」
    • 「把這些 if/else 改成 Strategy Pattern」
    • 「把命名改成符合專案規範（PascalCase / 私有 _ 開頭）」

建議下次有適合的時機，試著從「請分析」進階到「請改」。你的把關習慣 + Cursor 的 Accept/Reject 機制 = 安全又有效率。如果改出來品質不好可以馬上 reject，沒有任何負擔。

2. 用 @ 語法指定檔案，比截圖或打檔名更準

觀察到你目前指定要 AI 分析的檔案，主要用兩種方式：

  • 直接打檔名（例如「請分析 UIGVECardOverviewMain.cs」）— AI 要自己去搜尋，可能找錯版本或漏看
  • 截圖讓 AI 辨識 IDE 中的檔案 — AI 要從像素去推斷檔名，精度有限

這兩種方式 AI 都得「猜」你指的是哪個檔，分析時容易看不到完整內容。建議改用 Cursor 內建的 @ 語法：

怎麼用：在 Chat 輸入框打 @，會跳出檔案/資料夾的自動完成清單，選一個就會把整個檔案內容直接餵給 AI，不用 AI 自己去找。
範例：
@UIGVECardOverviewMain.cs 請分析這個類別的職責與資料流向
@AINpc.cs @AINpcManager.cs @AINpcSystem.cs 這三個的職責怎麼分配
還可以指定範圍：@UIGVECardOverviewMain.cs (1-100) 只餵 1-100 行。

趙健智和高偉傑都常這樣用，分析跟修改的精度會明顯提升。之後想試 Cursor 改 code 的時候，這個技巧更重要 — 沒有 @ 直接餵檔，AI 在實作時更容易找錯檔或只看到片段就動手。

3. 編譯錯誤交給 Unity MCP + 好一點的模型直接 debug

觀察到你貼編譯錯誤訊息給 AI 時，第一次的修法有時不到位（例如 BuildingPositionConfig 那個 CS0246，AI 第一次答「補 using」，但第二行同錯誤再貼，AI 才發現要新建型別）。這通常是兩個原因疊加：

  • AI 只看到單行訊息，沒看到 Unity 整體的錯誤上下文
  • 模型本身的推理深度不夠，第一次傾向給「最淺的猜測」
兩個立刻能改善的做法：
  • (a) 既然已經裝了 Unity MCP，請直接讓 AI 自己看錯誤：不用手貼，直接說「Unity 有編譯錯誤，請你看一下並修」即可。AI 透過 MCP 可以直接抓到 console 完整錯誤列表，比你逐行貼快很多，而且能一次看到「跨檔同根因」的全貌。
  • (b) 模型升級到 Claude Opus 4.7 或 Sonnet 4.6：這類較強的模型在第一次回答就傾向找根因，不會只解單行錯誤，反覆來回的次數會明顯減少，整體 token 反而更省。Cursor 右下角可切換模型，建議優先試這兩個。

💡 給紀閎耀

1. 拆解大任務 — AI 在小範圍上表現最好

觀察到部分任務的範圍比較大（例如「晝夜循環該做什麼」「掃所有紅點 prefab 找三種情況」），AI 一次處理會容易出錯或方向偏。

建議做法：把大任務拆成幾個小步驟，一個一個確認再往下。例如紅點系統可以拆成：
  1. 先請 AI 列出掃描的 plan（不寫 code），你 review 過 OK 才往下
  2. 請 AI 產核心掃描邏輯（單一資料夾、單一條件），跑過確認結果對
  3. 再請 AI 擴充其他條件 / 範圍
這樣每一步都是小範圍、可驗證的，AI 出錯的機率大幅降低。

2. 處理 AI 中文輸出亂碼

你提到「你怎麼產了一堆亂碼」，這是 AI 模型在長中文+程式碼混排時偶發的字元損壞。減少觸發：

  • 在 prompt 裡指定「程式碼註解請用英文，說明文字用簡短繁中」
  • 產出較長的工具腳本時，可以分段請求（先產 Python 版、再產 C# 版）
  • 出現亂碼時直接說「請用純 ASCII 重新輸出一次」就會修正

3. 第一次 prompt 就附實際結構範例

本週發現「祖先鏈邏輯」是因為實際 prefab 結構跟 AI 預設假設不同。

建議做法：在第一次 prompt 就附 1-2 個實際 prefab 的局部結構截圖或 YAML 片段，AI 就不會用錯誤假設出方案，可以省一輪糾正。

4. 「用 AI 造工具」的方向很對，可以更常用

你本週的紅點系統稽核工具是非常好的用法，這類「一次寫好可重複使用」的工作很適合：

  • 找 missing reference 的 prefab
  • 找命名不符規範的腳本
  • 批次重新命名/搬移資源
  • 找掛了 component 但已不被引用的物件

💡 給高偉傑

1. 你的「分析→選方案→執行」工作流非常好，請繼續保持

你在 BuildingManager.Floor.cs 的處理是團隊內最完整的 Agent 模式示範：分析 → 拿到 A/B 方案 → 選一個 → 執行。這個流程的好處是每一步你都有控制權，AI 不會自己偏掉。

2. 進階：改完後請 AI 再 review 自己的修改

效率型使用者最大的風險是漏掉 AI 的隱性錯誤。建議養成二次驗證習慣：

建議 prompt：「你剛剛改了 X 檔案，請 review 自己的修改，找出有沒有：(1) 沒處理到的邊界條件 (2) 跟原本邏輯衝突的地方 (3) 可能的 NRE 或記憶體問題」
特別在跨平台 (#if) 或生命週期相關的修改上很有用。

3. 跨檔分析可以一次餵多個 @

你已經會用 @檔名 (行數)，可以再進一步一次餵多個檔案：

@UIGamePassPanel.cs @GamePassManager.cs @GamePassSystem.cs 請分析這三者的職責分配是否合理，有沒有可以合併或拆分的地方

💡 給趙健智

1. 「先把需求問清楚再實作」— 避免 AI 改來改去

觀察到本週 96 輪對話中，有些需求反覆貼了多次（例如「員工不能在多個投資單」貼 2 次、「稀有度排序邏輯」貼 3 次），這通常是「AI 第一次理解不夠完整就動手改」造成的循環。

建議做法：複雜需求「先問清楚再實作」三步驟
  1. 第一段先說「我等下要做 X 功能，你覺得需要先問我哪些問題才能準確實作？」
  2. AI 會列出它的疑問，你一次回答完
  3. 確認 AI 真的理解了，再說「OK，請實作」
這個方式可以把「邊做邊修」的循環，變成「想清楚一次做對」。

2. 用 Cursor Rules / AGENTS.md 管理長期規則

觀察到你需要在對話中重複強調的「專案規則」（例如「員工不能在多個投資單」「稀有度排序由高到低」「Timer 從 OrgPanel 切回 Panel 時不要重啟」），其實可以寫成專案級長期規則。

建議做法：在專案根目錄建立 AGENTS.md，把這類規則寫進去。每次新對話 AI 都會自動讀取，不用重貼。
範例內容： 
# Invest 系統規則
- 一個員工同時間只能在一個投資單
- 稀有度顯示順序：由高到低（SSR > SR > R）
- Timer 從 InvestOrgPanel 切回 InvestPanel 時不要重新計時
- 假伺服器邏輯放在 InvestFakeServer.cs，UI 層不直接接觸假資料

3. 大功能完成就開新對話

單段 96 輪對話雖然展現了你的深度使用能力，但 AI 在長對話中會慢慢忘記早期決策。建議：

  • 每完成一個獨立功能（例如「編制系統」做完後）就開新對話
  • 同樣的需求/糾正講第 2 次以上 → 該開新對話了
  • 新對話開頭可以請 AI 先「總結目前已完成 + 重要決策」當作 context

4. Plan 模式的進階用法

你已經會用 Plan 模式很棒，可以再進一步：

  • 大功能開發前先讓 AI 出 plan（不執行）
  • 自己 review plan，覺得方向對才按 Implement
  • 覺得 plan 漏了什麼，先補完整再實作

通用技巧 — 大家都可以試試看

🔧 技巧 1：Cursor Rules / AGENTS.md — 專案級長期記憶

把「整個專案都要遵守的規則」寫到 .cursor/rules/ 或專案根目錄的 AGENTS.md，AI 每次都會自動讀取，不用每次重貼。

適合放進去的內容：

  • 專案命名規則（例如「Manager 命名規則」「UI 前綴」）
  • 禁用 API（例如「不要用 SendMessage」「Update 裡禁止 GetComponent」）
  • 架構約束（例如「Manager 只透過 System 跟 server 溝通」）
  • 領域知識（例如「IsIntoStore 在門外也可能是 true」）
  • 業務規則（例如「員工不能同時在多個投資單」）

🔧 技巧 2：@ 語法 — 精準指定檔案

在 Cursor chat 中用 @檔名 可以把整個檔案內容直接餵給 AI，比讓 AI 自己搜尋更精準快速。

@UIInvestPanel.cs (1-100) 請分析這段
@BuildingManager.Floor.cs @BuildingFloor.cs 這兩個腳本的關聯

🔧 技巧 3：「先問再做」工作流 — 避免 AI 改來改去

複雜或模糊的需求，可以先讓 AI 確認理解再實作：

萬用 prompt：「我等下要做 X 功能，你覺得需要先問我哪些問題才能準確實作？」
AI 會列出它的疑問清單，你一次回答完，比邊做邊改省時間很多。

🔧 技巧 4：任務拆分 — 從小步驗證開始，需求清楚後再放大

剛開始用 AI 寫實作任務時，建議先從小範圍、可驗證的步驟練手感：

  1. 列計畫（不寫 code）— 你 review 方向
  2. 產核心邏輯（單一情境）— 跑過確認結果
  3. 擴充其他情境
  4. 邊界條件處理（null 檢查、錯誤處理等）

每一步都可驗證，AI 出錯的機率大幅降低，這是最穩的入門方式。

常見迷思：「我把整個大功能 prompt 一次丟給 AI 它就幫我做完了」
如果需求模糊、或同樣需求有很多種完全不同的合理實作，一口氣丟大功能會有兩個問題：
(1) 出來的東西通常跟自己想的不一樣，要花更多時間修改
(2) Token 也會一口氣噴光，做到一半額度沒了反而尷尬
所以「需求還沒想清楚時，先小步堆疊」是最安全的做法。
進階：當需求夠明確時，範圍可以漸進式放大
不需要永遠停在「一次只改一個 function」，當你越來越熟悉 AI 的脾氣後，可以慢慢把範圍從
function → 整個檔案 → 整個模組 → 多檔案重構 這樣往上加。

判斷「可以放大」的兩個條件：
(1) 需求很明確，你自己能清楚講出「成功的樣子長什麼樣」
(2) 同樣需求不會有完全不同的合理解法（例如「把這 5 個 if/else 改成 Strategy Pattern」就很明確；「優化這個系統」就太模糊）

另外，現在較新的模型（Opus 4.7 / Sonnet 4.6）的 Plan 模式會主動問你不確定的點，sub-agent / 任務分解能力也越來越強，越能放心放大範圍 — 反而更快、更省 token。

🔧 技巧 5：模型選擇 — 用好一點的模型反而更省 token

很多人會擔心「用強模型 token 會被噴光」，所以選便宜的模型。實際上常常相反：

  • 弱模型容易給「淺層猜測」，第一次答不到位 → 你貼新的錯誤 → 再答一次 → 來回 3-5 次才解決
  • 強模型第一次就找根因 → 1-2 次解決

反覆來回的總 token 成本，常常比直接用強模型還高。建議優先嘗試的模型：

推薦順序：
  • Claude Opus 4.7 — 最強的推理跟 plan 能力，複雜重構/架構分析時最有感
  • Claude Sonnet 4.6 — 速度跟成本的甜蜜點，日常實作、debug、寫 code 都很夠用
這兩個對 Unity / C# 的理解都很到位，一般情境就足夠。Cursor 右下角的模型選單可以隨時切換。

🔧 技巧 6：開新對話的時機

長對話 AI 容易遺忘早期決策、犯重複的錯。建議在以下時機開新對話：

  • 完成一個獨立功能模組
  • 同樣的需求/糾正講第 2 次以上
  • 主題切換（例如從 Invest 系統跳到 Plot 系統）
  • 對話超過 30-50 輪

開新對話前可以請 AI 「總結目前已完成 + 待辦 + 重要決策」，貼到新對話開頭當 context。

🔧 技巧 7：給 AI 領域知識作為前提條件

在第一次 prompt 就主動提供：

  • 專案特有術語的解釋
  • 已知的約束條件（例如「不要動到事件訂閱邏輯」）
  • 不希望被改動的部分
  • 實際資料的結構範例（截圖或片段）

這樣可以避免 AI 用錯誤假設出方案，省下糾正的輪數。
